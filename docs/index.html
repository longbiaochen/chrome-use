Managed CfT Runtime

Fast inspect, stable auth, direct CDP control.

chrome-use keeps a managed Chrome for Testing browser alive for coding agents, then turns human clicks and real login flows into structured, reusable browser state.

Open the repo Read the architecture
Why this release

Chrome 136+ no longer accepts --remote-debugging-port on the user's default profile. This repo now defaults to a managed Chrome for Testing runtime instead of an app shim or direct attach to Google Chrome.app Default.

Runtime contract
  • Managed browser under ~/.chrome-use/browsers/chrome-for-testing
  • Dedicated browser data under ~/.chrome-use/browser-data/<channel-or-version>
  • Single debug endpoint at 127.0.0.1:9223
Public skills

chrome-inspect for live selection handoff. chrome-auth for direct-CDP auth flows.

What stays strong

Operators still get one persistent browser world for inspect and auth. Agents still get direct CDP, stable cookies, reusable targets, and target-bound routing.

What changed

No more copied app bundle, renamed Chrome shim, or Dock-level packaging burden. The installer now downloads Chrome for Testing from the official feed by default.

What to expect

This is an automation browser, not your day-to-day Chrome profile. It does not inherit your existing sync state, extensions, or user cookies.

chrome-inspect

Click once on the real page. Get structured DOM context back in the same turn.

  • Persistent in-page toolbar
  • Durable latest-selection recovery
  • Target-bound workflow routing

chrome-auth

Use direct CDP to find the real auth surface and finish login without screenshot guesswork.

  • Page listing and binding
  • DOM and accessibility snapshots
  • Waits and state-aware auth control

Quick start

The default install path downloads and prepares managed Chrome for Testing automatically.

git clone https://github.com/longbiaochen/chrome-use
cd chrome-use
bash install/install.sh

Prefer the full repo docs for install details, runtime boundaries, and client-specific notes.
